How the Web Works
HTTP request/response, DNS, browsers, servers, clients
🔓
⌄

The Web at 30,000 Feet

Every time you visit a website, a complex series of events happens in milliseconds. Understanding this process makes you a better developer — you can debug problems faster, design better systems, and communicate clearly with other engineers.

The Request-Response Cycle

  1. You type a URL in the browser: https://paystack.com
  2. Browser asks DNS to translate the domain to an IP address
  3. Browser makes an HTTP request to that IP address on port 443 (HTTPS)
  4. Server processes the request and sends back an HTTP response with the HTML
  5. Browser parses the HTML, makes additional requests for CSS, JS, and images
  6. Browser renders the complete page

DNS — The Phone Book of the Internet

DNS (Domain Name System) translates human-readable domains (paystack.com) to machine-readable IP addresses (13.53.47.21). Without DNS you would need to remember IP addresses instead of names. DNS lookup takes milliseconds and results are cached.

HTTP Methods

  • GET — Retrieve a resource. No body. Idempotent (safe to repeat).
  • POST — Create a new resource. Has a body with the data.
  • PUT — Replace a resource completely.
  • PATCH — Partially update a resource.
  • DELETE — Remove a resource.

HTTP Status Codes

  • 2xx — Success: 200 OK, 201 Created, 204 No Content
  • 3xx — Redirect: 301 Moved Permanently, 302 Temporary Redirect
  • 4xx — Client error: 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found
  • 5xx — Server error: 500 Internal Server Error, 503 Service Unavailable

HTML — The Skeleton of the Web

HTML (HyperText Markup Language) defines the structure and content of a web page. HTML5 introduced semantic elements that make the meaning of content explicit — both for browsers, screen readers, and search engines.

Semantic HTML

Semantic elements clearly describe their purpose. Use them instead of generic divs whenever possible.

<header>Site header</header>
<nav>Navigation links</nav>
<main>
  <article>A self-contained piece of content</article>
  <section>A thematic grouping</section>
  <aside>Related sidebar content</aside>
</main>
<footer>Site footer</footer>

Forms

<form action="/submit" method="POST">
  <label for="email">Email:</label>
  <input type="email" id="email" name="email" required/>
  
  <label for="password">Password:</label>
  <input type="password" id="password" name="password" required/>
  
  <button type="submit">Login</button>
</form>

Accessibility Basics

  • Always use alt text on images: <img src="..." alt="Description of image"/>
  • Associate labels with inputs using for and id attributes
  • Use semantic elements — screen readers understand nav, main, button
  • Ensure sufficient colour contrast for readability
  • Make all interactive elements keyboard accessible

SEO Basics in HTML

  • <title> — Shown in browser tab and search results. Most important.
  • <meta name="description"> — Shown in search snippets. Affects click rate.
  • One <h1> per page — Your main heading. Use heading hierarchy correctly.
  • Descriptive URL structure and internal links

CSS — Styling the Web

CSS (Cascading Style Sheets) controls the visual presentation of HTML. Modern CSS has powerful layout tools — Flexbox and Grid — that replaced decades of float-based layout hacks. Together with responsive design techniques, they enable beautiful layouts that work on any screen size.

Flexbox — One-Dimensional Layouts

/* Apply to the parent container */
.nav {
  display: flex;
  justify-content: space-between;  /* horizontal alignment */
  align-items: center;             /* vertical alignment */
  gap: 1rem;
}

/* Common justify-content values */
/* flex-start | center | flex-end | space-between | space-around */

CSS Grid — Two-Dimensional Layouts

.card-grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr);  /* 3 equal columns */
  gap: 1.5rem;
}

/* Responsive: 1 column on mobile, 3 on desktop */
.card-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(280px, 1fr));
  gap: 1.5rem;
}

Responsive Design with Media Queries

/* Mobile first: base styles apply to all screens */
.hero { font-size: 1.5rem; padding: 1.5rem; }

/* Add styles for larger screens */
@media (min-width: 768px) {
  .hero { font-size: 2.5rem; padding: 3rem; }
}

@media (min-width: 1024px) {
  .hero { font-size: 3.5rem; padding: 5rem; }
}

CSS Variables

:root {
  --primary: #00e5ff;
  --bg: #060810;
  --text: #e8edf5;
}

button { background: var(--primary); color: var(--bg); }

JavaScript on the Server

Node.js runs JavaScript outside the browser — on a server. This allows JavaScript developers to write both frontend and backend code in the same language, sharing utilities and reducing context-switching.

npm — Node Package Manager

# Initialise a new project
npm init -y

# Install packages
npm install express
npm install --save-dev nodemon

# package.json scripts
{
  "scripts": {
    "start": "node server.js",
    "dev": "nodemon server.js"
  }
}

Building a REST API with Express

const express = require("express");
const app = express();

app.use(express.json());  // Parse JSON request bodies

// Routes
app.get("/api/users", async (req, res) => {
  const users = await db.query("SELECT * FROM users");
  res.json(users);
});

app.post("/api/users", async (req, res) => {
  const { name, email } = req.body;
  const user = await db.create({ name, email });
  res.status(201).json(user);
});

app.listen(3000, () => console.log("Server running on port 3000"));

Middleware

Middleware functions run between receiving a request and sending a response. They can: authenticate requests, log activity, parse request bodies, handle CORS, and rate limit.

// Authentication middleware
const auth = (req, res, next) => {
  const token = req.headers.authorization;
  if (!isValidToken(token)) return res.status(401).json({ error: "Unauthorised" });
  next();  // Pass to next handler
};

app.use("/api/protected", auth);

Every Web App Needs Data Storage

Most web applications store data in databases. Understanding SQL and database design is fundamental — even if you use an ORM (which abstracts the SQL), understanding what happens underneath makes you a better developer.

Relational Database Concepts

  • Table — Data organised in rows and columns (like a spreadsheet)
  • Primary key — A unique identifier for each row (usually id)
  • Foreign key — Links one table to another
  • Index — Speeds up queries on frequently searched columns
  • Transaction — A group of operations that succeed or fail together

Essential SQL

-- Create table
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(255) UNIQUE NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Query
SELECT * FROM users WHERE active = true ORDER BY name LIMIT 20;

-- Join tables
SELECT orders.id, users.name, orders.total
FROM orders
JOIN users ON orders.user_id = users.id
WHERE orders.status = 'pending';

-- Insert
INSERT INTO users (name, email) VALUES ('Israel', 'i@example.com');

-- Update
UPDATE users SET name = 'New Name' WHERE id = 5;

-- Delete
DELETE FROM users WHERE id = 5;

ORMs — Object-Relational Mappers

ORMs like Prisma (Node.js) and SQLAlchemy (Python) let you interact with databases using your programming language instead of raw SQL. They handle schema migrations, type safety, and query building.

Authentication in Web Applications

Authentication verifies who a user is. Session management maintains that verified identity across multiple HTTP requests (since HTTP is stateless — each request is independent). These are among the most security-critical parts of any web application.

Session-Based Authentication

  1. User logs in with credentials
  2. Server verifies credentials and creates a session stored in the database
  3. Server sends a session ID in a cookie to the browser
  4. Browser sends the cookie with every subsequent request
  5. Server looks up the session ID to identify the user

JWT (JSON Web Tokens)

JWTs are self-contained tokens that contain encoded claims about the user. The server signs the token with a secret key — the signature can be verified without a database lookup.

// Token structure: header.payload.signature (base64 encoded)
// Payload contains: { userId: 123, role: 'admin', exp: 1735689600 }

// Server signs token on login
const token = jwt.sign({ userId: user.id, role: user.role }, SECRET_KEY, 
                         { expiresIn: '24h' });

// Client sends: Authorization: Bearer <token>
// Server verifies:
const decoded = jwt.verify(token, SECRET_KEY);

Password Hashing

const bcrypt = require("bcrypt");

// When creating account
const hash = await bcrypt.hash(plainPassword, 12);  // Store hash, not password
await db.saveUser({ email, passwordHash: hash });

// When logging in
const match = await bcrypt.compare(plainPassword, storedHash);
if (!match) return res.status(401).json({ error: "Invalid credentials" });

OAuth 2.0 — Login with Google/GitHub

OAuth allows users to authenticate with a trusted third party (Google, GitHub) without sharing their password with your app. Your app receives an access token to identify the user.

Building and Consuming APIs

Modern web development is built on APIs. Your frontend calls your backend API. Your backend calls third-party APIs (payment, email, SMS). Understanding API design principles makes you a better builder and a better integrator.

REST API Design Best Practices

  • Use nouns for resources — /api/users not /api/getUsers
  • Use HTTP methods correctly — GET to read, POST to create, PUT/PATCH to update, DELETE to remove
  • Return appropriate status codes — 201 for created, 404 for not found, 400 for validation errors
  • Version your API — /api/v1/users so you can make breaking changes later
  • Use pagination — /api/users?page=2&limit=20 for large datasets
  • Return consistent error format — {"error": "Not found", "code": "USER_NOT_FOUND"}

Webhooks

Webhooks are reverse APIs — instead of your app calling a service, the service calls your app when something happens. Paystack sends a webhook to your server when a payment is completed. Your server handles it and updates the user's account.

// Express webhook endpoint
app.post("/webhook/paystack", (req, res) => {
  const event = req.body;
  if (event.event === "charge.success") {
    await activateUserAccount(event.data.metadata.userId);
  }
  res.sendStatus(200);  // Always respond quickly
});

GraphQL

An alternative to REST where clients specify exactly what data they need in a single query. Reduces over-fetching and under-fetching. Used by GitHub, Shopify, and many large APIs.

// Glossary Webhook — A callback URL that receives real-time notifications from external services

Getting Your App to the World

Building a web application is only half the job. Deploying it reliably and maintaining it in production is equally important. Modern deployment uses automation to eliminate manual steps and human error.

Environment Variables

Never hardcode secrets (API keys, database passwords) in code. Use environment variables and a .env file locally.

# .env file (never commit this to git!)
DATABASE_URL=postgresql://user:password@localhost:5432/mydb
API_KEY=sk-abc123
JWT_SECRET=super-secret-string

# In Node.js
require('dotenv').config();
const db = process.env.DATABASE_URL;

GitHub Actions — CI/CD

# .github/workflows/deploy.yml
name: Deploy
on:
  push:
    branches: [main]

jobs:
  test-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '20'
      - run: npm install
      - run: npm test
      - run: npm run deploy  # Only runs if tests pass

Hosting Options

  • GitHub Pages — Free, for static sites only
  • Vercel — Best for frontend and serverless functions. Free tier is generous.
  • Railway — Easy Node.js and database hosting. Generous free tier.
  • Render — Full-stack apps and databases. Free tier available.
  • AWS / Google Cloud / Azure — Enterprise grade. More complex but most powerful.

Why Performance Matters

Web performance directly affects user experience and business results. Google uses Core Web Vitals as a ranking signal — slow pages rank lower. A 1-second delay in mobile load time reduces conversions by up to 20%. In Nigeria and across Africa where many users are on 3G with limited data plans, performance is not a luxury — it is accessibility.

Core Web Vitals

  • LCP (Largest Contentful Paint) — How long until the largest visible element loads. Target: under 2.5 seconds.
  • FID/INP (Interaction to Next Paint) — How quickly the page responds to user interaction. Target: under 200ms.
  • CLS (Cumulative Layout Shift) — How much the page layout unexpectedly shifts. Target: under 0.1.

Performance Techniques

  • Lazy loading — Load images only when they scroll into view: <img loading="lazy"/>
  • Image optimisation — Use WebP format. Serve correct sizes. Compress images.
  • Code splitting — Only load JavaScript needed for the current page
  • Caching — Cache-Control headers tell browsers how long to cache resources
  • CDN — Serve static assets from servers geographically close to users
  • Minification — Remove whitespace and comments from CSS and JS files

Lighthouse

Google's free tool that audits your site for Performance, Accessibility, Best Practices, and SEO. Run it in Chrome DevTools → Lighthouse or at web.dev/measure. The report shows exactly what to fix and why.

Open Graph Tags for Social Sharing

<meta property="og:title" content="Your Page Title"/>
<meta property="og:description" content="Page description"/>
<meta property="og:image" content="https://yoursite.com/image.jpg"/>
